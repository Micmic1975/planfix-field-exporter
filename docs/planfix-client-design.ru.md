# Проектирование безопасного клиента Planfix API

Цель шага: описать, как должен быть устроен слой HTTP-запросов к Planfix перед
добавлением поиска использований полей в формулах.

Документ не вводит новые endpoints и не меняет правила из
`inputs/swagger.json`. Все методы, параметры и поля должны быть подтверждены
`inputs/swagger.json`, текущим кодом проекта или реальным read-only ответом
Planfix.

## Задачи клиента

Клиент Planfix API должен:

- выполнять все `GET` и `POST` запросы через одну общую точку;
- выдерживать паузу минимум `1.25` секунды между любыми HTTP-запросами;
- одинаково работать для GUI, консольной версии и read-only скриптов;
- возвращать понятные ошибки для пользователя;
- не печатать и не сохранять токен доступа;
- корректно сохранять кириллицу в JSON/CSV/output-файлах;
- давать прогресс для долгих операций.

## Rate limit

Правило: между завершением подготовки одного HTTP-запроса и отправкой
следующего HTTP-запроса должна быть пауза минимум `1.25` секунды.

Практическая реализация:

- хранить время последнего HTTP-запроса в общем объекте клиента;
- перед каждым новым запросом вычислять, сколько осталось до `1.25` секунды;
- если осталось больше нуля, делать `sleep`;
- обновлять `last_request_at` непосредственно перед отправкой запроса;
- использовать один и тот же клиент внутри одной долгой операции.

Для GUI запросы должны выполняться не в основном Tkinter-потоке. Rate limiter
при этом остается внутри worker-клиента.

## Ошибки сети и API

Клиент должен отдельно обрабатывать:

- таймаут;
- отсутствие сети;
- DNS/SSL ошибки;
- HTTP-статус не `200`;
- ответ Planfix с `result != "success"`;
- ответ, который не удалось разобрать как JSON;
- неожиданный формат JSON, например отсутствие массива `customfields`.

Сообщения для пользователя должны быть на русском. В технический output можно
добавлять детали: метод, path, HTTP-статус, текст ошибки Planfix и фрагмент
ответа.

Токен нельзя включать:

- в текст ошибки;
- в лог;
- в JSON output;
- в traceback, показываемый пользователю.

## Прогресс долгих операций

Для долгих операций нужен общий callback прогресса.

Минимальные поля события прогресса:

- `stage` — текущий этап, например `Загрузка списка шаблонов задач`;
- `entity_type` — тип сущности Planfix: `задача`, `шаблон задачи`, `контакт`,
  `проект`, `сотрудник`, `справочник`, `аналитика`, `основное поле`;
- `processed_pages` — сколько страниц обработано;
- `processed_entities` — сколько сущностей обработано;
- `last_request_at` — время последнего HTTP-запроса;
- `output_file` — текущий файл результата, если он уже создан;
- `message` — короткое человекочитаемое сообщение.

В консольном скрипте это можно печатать строками. В GUI это должно попадать в
status bar и, возможно, в отдельное поле журнала.

## Работа с кириллицей

Planfix-данные почти всегда содержат русские названия полей и сущностей, поэтому
нужно явно соблюдать кодировки:

- читать и писать JSON только с `encoding="utf-8"`;
- сохранять JSON с `ensure_ascii=False`;
- CSV для Excel сохранять как `utf-8-sig`;
- не делать ручное URL-кодирование параметров, если используется `requests`
  через `params`;
- не сравнивать русские названия как единственный идентификатор поля;
- для поиска поля в формуле использовать id и подтвержденный синтаксис ссылки,
  а название использовать только как дополнительную эвристику после проверки.

## Предлагаемая структура кода

Минимально инвазивный вариант:

- оставить публичные функции `planfix_get`, `planfix_post`,
  `get_object_list`, `get_task_template_list` для совместимости;
- добавить класс `PlanfixClient`;
- внутри `planfix_get` и `planfix_post` использовать общий экземпляр или
  вспомогательную функцию с rate limiter;
- для долгих скриптов создавать отдельный `PlanfixClient` с progress callback.

Целевая форма:

```text
planfix_client.py
  PlanfixClient
    request(method, path, params=None, payload=None)
    get(path, params=None)
    post(path, payload=None)
    load_all_pages(fetch_page, items_key, page_size=100)

  PlanfixApiError
  PlanfixNetworkError
  PlanfixResponseFormatError

  совместимые функции:
    planfix_get(config, path, params=None)
    planfix_post(config, path, payload=None)
```

## Совместимость с текущим приложением

На первом техническом шаге нельзя ломать существующие сценарии:

- проверка настроек;
- загрузка списка объектов задач;
- загрузка списка шаблонов задач;
- просмотр пользовательских полей объекта задачи;
- просмотр пользовательских полей шаблона задачи;
- выгрузка CSV.

Поэтому внедрение клиента лучше делать в два этапа:

1. Внутренняя замена HTTP-слоя без изменения внешних функций.
2. Перевод новых read-only скриптов и будущего поиска формул на
   `PlanfixClient`.

## Что не делаем на этом шаге

На этом шаге не нужно:

- выполнять реальные запросы к Planfix;
- добавлять новые endpoints;
- предполагать синтаксис формул;
- реализовывать поиск использований;
- менять GUI.

## Ожидаемый результат следующего шага

Следующий шаг должен добавить read-only разведочный скрипт, который:

- читает только endpoints и поля, подтвержденные `inputs/swagger.json` и этим
  документом;
- использует общий rate limit `1.25` секунды;
- регулярно показывает прогресс;
- сохраняет ответы в `output/`;
- не выводит токен;
- помогает понять, где Planfix реально возвращает `formula` и как выглядят
  ссылки на поля внутри формул.
