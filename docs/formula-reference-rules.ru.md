# Правила поиска использований поля в формулах Planfix

Источник фактов:

- `inputs/swagger.json`;
- реальные read-only ответы Planfix из
  `output/planfix_probe/20260531_172845`;
- текущий код проекта.

## Что подтверждено реальными ответами

Во втором разведочном проходе с `--include-formula-probe` Planfix вернул
формулы для вычисляемых пользовательских полей задач и контактов.

Подтвержденные результаты:

- пользовательские поля задач: `898` полей, `70` вычисляемых, у всех `70`
  есть `formula`;
- пользовательские поля контактов: `196` полей, `28` вычисляемых, у всех `28`
  есть `formula`;
- пользовательские поля проектов: `54` поля, вычисляемых полей не найдено;
- пользовательские поля сотрудников: `12` полей, вычисляемых полей не найдено;
- основные пользовательские поля: `11` полей, вычисляемых полей не найдено;
- в малой scoped-выборке справочников и аналитик вычисляемых полей не найдено.

Дополнительный targeted probe по справочнику `9416` подтвердил еще один
источник формул:

- справочник `9416` = `Предупреждения в задачах и контактах`;
- запись `52` успешно читается через `GET /directory/9416/entry/52`;
- в справочнике есть текстовые поля:
  - `24228` = `Формула`, тип `2` / `Текст`;
  - `24510` = `Комментарии`, тип `2` / `Текст`;
- оба текстовых поля могут содержать ссылки вида `{{...}}`;
- значение текстового поля приходит как HTML, внутри которого формульный текст
  и ссылки Planfix могут быть закодированы HTML-сущностями.

Это означает: источниками поиска должны быть не только вычисляемые поля задач и
контактов, но и текстовые поля записей справочников, если в них есть ссылки
`{{...}}`.

## Подтвержденный синтаксис ссылок

Ссылки на поля в формулах находятся внутри двойных фигурных скобок:

```text
{{Задача.Название поля}}
{{Контакт.Название поля}}
```

Также подтверждены вложенные ссылки:

```text
{{Задача.Проект.Контрагент}}
{{Контакт.Проект контрагента."Количество сделок. Всего"}}
{{Задача.Компания клиента.Проект контрагента."Проекты контрагентов / Количество активных сделок. УЦ"}}
```

Подтвержденные первые сегменты ссылок:

- `Задача`
- `Контакт`
- `Справочник`
- `Аналитика`
- `Системные`

## Кавычки и HTML-сущности

Planfix может оборачивать сегмент пути в кавычки, если внутри названия есть
точка или другой сложный текст:

```text
{{Задача."Сделка / Маржа, руб."}}
```

В реальных формулах часть кавычек пришла как HTML-сущности:

```text
{{Задача.&quot;Количество активных подзадач. Тендерные процедуры&quot;}}
```

В текстовых полях справочников весь текст может приходить как HTML:

```text
<p>%%$ЕСЛИ({{Задача.Тип сделки}}=&quot;&quot;)$%%...</p>
```

Правило реализации:

- перед разбором ссылки нужно применять `html.unescape`;
- после этого `&quot;` становится `"`, а `&#39;` становится апострофом;
- делить путь по точкам можно только вне кавычек.

## Почему нельзя искать простой подстрокой

Простой поиск по названию поля даст ложные совпадения.

Примеры рисков:

- поле `Проект` совпадет с частью `Проект контрагента`;
- поле `ИНН` может встретиться внутри другого названия;
- поле `Количество сделок. Всего` содержит точку, и без учета кавычек путь
  будет неправильно разбит;
- название поля может встречаться в строковом литерале формулы, а не в ссылке
  `{{...}}`;
- в формулах используются русские функции и текстовые строки, например
  `ЕСЛИ`, `ИЛИ`, `СОДЕРЖИТ`.

Поэтому основной поиск должен идти только по разобранным ссылкам внутри
`{{...}}`.

## Нормализация текста

Для сравнения сегментов ссылки и названий полей нужно:

- применять `html.unescape`;
- заменять `\r\n` и `\r` на `\n`;
- схлопывать повторные пробелы;
- обрезать пробелы по краям;
- снимать внешние двойные кавычки у сегмента, если весь сегмент ими обернут;
- не менять регистр по умолчанию, потому что русские названия в Planfix обычно
  чувствительны для человека; для дополнительной эвристики можно использовать
  `casefold`, но не как основной критерий.

## Разбор ссылки

Алгоритм разбора одной ссылки:

1. Найти все фрагменты формулы по регулярному выражению `{{...}}`.
2. Для каждого фрагмента применить `html.unescape`.
3. Убрать внешние пробелы.
4. Разделить строку на сегменты по точке `.` вне двойных кавычек.
5. Для каждого сегмента применить нормализацию.
6. Получить путь вида:

```text
["Задача", "Компания клиента", "Проект контрагента", "Количество активных сделок. УЦ"]
```

Если кавычки не сбалансированы, ссылку не нужно отбрасывать полностью. Ее
следует сохранить как "неразобранную ссылку" и показать пользователю как
сомнительное совпадение только при ручной диагностике.

## Первичный алгоритм поиска выбранного поля

Для выбранного поля известны как минимум:

- `id`;
- `name`;
- `names`;
- `type`;
- `objectType`;
- тип сущности Planfix: задача, контакт, проект, сотрудник, справочник,
  аналитика, основное поле.

В наблюдаемых формулах ссылки идут по названиям, а не по `id`. Поэтому `id`
нужен для идентификации выбранного поля в приложении и output-файлах, но не
является основным способом поиска внутри формулы.

Основной критерий совпадения:

- выбранное название поля совпадает с одним из сегментов разобранной ссылки
  после первого сегмента;
- совпадение должно быть полным по сегменту, а не подстрочным.

Пример:

```text
Выбрано поле: Проект контрагента
Ссылка:        {{Контакт.Проект контрагента."Количество сделок. Всего"}}
Результат:     совпадение по сегменту 2
```

Дополнительный критерий для вложенных полей:

- если пользователь выбрал поле `Количество сделок. Всего`, ссылка выше тоже
  должна считаться использованием, потому что это последний сегмент пути.

Дополнительный подтвержденный случай для полей сделки/аналитики:

- Planfix может писать ссылку на поле через сегмент с разделителем `/`,
  например `{{Задача.Сделка / Сумма продажи с учетом скидок, с НДС, в валюте}}`;
- если выбранное поле совпадает с частью сегмента после `/`, это тоже
  считается использованием;
- это не простой поиск подстроки: проверяются только нормализованные части
  сегмента, разделенные `/`.

## Уровни уверенности совпадения

Результат поиска лучше хранить с уровнем уверенности.

### `exact_segment`

Полное совпадение нормализованного названия выбранного поля с сегментом ссылки.
Это основной надежный результат.

### `exact_quoted_segment`

То же самое, но сегмент был явно в кавычках. Это тоже надежный результат,
особенно для названий с точками.

### `casefold_segment`

Совпадение найдено только после `casefold`. Это полезно как дополнительная
подсказка, но его лучше отмечать отдельно.

### `exact_slash_part`

Полное совпадение нормализованного названия выбранного поля с частью сегмента,
разделенной `/`.

Пример:

```text
Выбрано поле: Сумма продажи с учетом скидок, с НДС, в валюте
Ссылка:        {{Задача.Сделка / Сумма продажи с учетом скидок, с НДС, в валюте}}
Результат:     совпадение по части сегмента 1
```

### `casefold_slash_part`

То же самое, но совпадение найдено только после `casefold`.

### `raw_substring`

Простое вхождение названия в сырой текст формулы. Это не надежное совпадение.
Его можно использовать только в диагностическом режиме, но не в основной
таблице результатов по умолчанию.

## Что показывать в результате

Для каждого найденного использования нужно сохранять:

- тип источника формулы: `задача`, `контакт`, `шаблон задачи`, и т.п.;
- для справочников: id справочника, название справочника, ключ записи,
  название записи, id текстового поля и название текстового поля;
- id источника, если есть;
- название источника, если есть;
- id вычисляемого поля;
- название вычисляемого поля;
- `objectType` вычисляемого поля;
- полный текст формулы;
- исходный фрагмент `{{...}}`;
- разобранные сегменты ссылки;
- индекс сегмента, где найдено совпадение;
- уровень уверенности совпадения;
- id и название выбранного поля.

## Ограничения, которые остаются

- Swagger не документирует `formula`; доступность подтверждена реальным API для
  задач и контактов.
- Формулы проектов, сотрудников, аналитик и основных полей не подтверждены,
  потому что в текущей выборке вычисляемых полей там нет.
- В справочниках формульные конструкции могут храниться не как `formula`
  пользовательского поля, а как значение текстового поля записи справочника.
- Поле справочника не обязано называться `Формула`, поэтому поиск по
  справочникам должен проверять все текстовые поля (`type == 2`) выбранных
  справочников или всех справочников, если пользователь запускает полный поиск.
- Ссылки используют названия, а не id, поэтому переименование поля в Planfix
  может влиять на поиск по старым формулам.
- Если два поля в разных сущностях имеют одинаковое название, нужен контекст
  пути (`Задача`, `Контакт`, `Справочник`, `Аналитика`) и ручная проверка
  неоднозначных случаев.

## Вывод для реализации

Первая реализация должна:

- искать только внутри `{{...}}`;
- декодировать HTML-сущности;
- разбирать путь по точкам с учетом кавычек;
- сравнивать полные сегменты, а не подстроки;
- сохранять уровень уверенности;
- показывать пользователю исходную ссылку и полную формулу;
- не полагаться на `id` поля внутри формулы, пока API не подтвердил такой
  синтаксис.
- дополнительно искать в текстовых полях записей справочников, потому что они
  могут содержать формулы/шаблоны с переменными Planfix.
